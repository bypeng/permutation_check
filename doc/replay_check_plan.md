# 專案追加計畫：Final Mission 賽事歷史重播系統 (Replay Dashboard)

## 1. 開發背景與目標
- **背景**：決賽（最終任務）圓滿結束，後端試算表已完整記錄了各隊在每一次驗收時的完整 Log 軌跡（時間戳、隊伍狀態、木片序列、LIS 長度、驗收次數）。
- **目標**：將現有的 `live_dashboard` 改裝為歷史重播專用的 `replay_check.html`，讓參賽者能透過動態時間軸（Time Slider）自由回溯決賽戰況，提供寫賽後心得與遊記的「精準史實依據」，進而提升社群遊記的產出質量與精彩度。

## 2. 核心功能設計

### 核心 A：動態時間軸控制項 (Time Slider)
- **UI 介面**：於網頁頂端設計一個滑桿（`<input type="range">`）與控制按鈕（▶️ 播放 / ⏸️ 暫停）。
- **刻度設計**：
    - 最小刻度 `0`：代表決賽正式開始（或第一筆 Log 時間）。
    - 最大刻度 `N-1`：代表最後一組通關（或最後一筆 Log 時間）。
- **聯動效果**：拖曳滑桿時，下方所有隊伍的卡片會立刻「時光倒流」或「快進」到該時間點的戰況。

### 核心 B：絲滑 FLIP 排序動畫繼承
- **視覺體驗**：完整保留並優化原有的 FLIP 動畫機制。當參賽者拉動時間軸時，隊伍卡片會隨著該時間點的排名與 LIS 長度，在畫面上順暢地上下滑動、即時跳色變色。
- **重現高潮**：完美重現決賽現場各隊伍「黃金交叉」、「卡關」、「極速衝刺」的動態名場面。

### 核心 C：遊記神器 ──「複製我的歷史軌跡」按鈕
- **貼心功能**：在畫面上提供一個一鍵複製（Copy to Clipboard）的功能。
- **動態產生文字**：當參賽者鎖定自己隊伍並拉到特定時間點時，點擊按鈕可直接產生文字格式，例如：
    > 📝 *【Final Mission 歷史軌跡回溯】*
    > *在決賽第 42 分鐘，我們隊伍成功進行了第 15 次驗收！此時 LIS 長度達到 38 片，位居全場第二！當時與第一名的差距僅剩 2 片！*
- **效益**：大幅降低參賽者撰寫回憶錄的阻力。

## 3. 技術實作架構

### 後端調整 (Code.gs)
- **新增 API 路由**：`?p=getReplayData`。
- **實作 `getReplayLogs()`**：
    1. 開啟並讀取「驗收紀錄工作表（Log Sheet）」。
    2. 撈取所有欄位：`Timestamp`, `TeamID`, `TeamName`, `CurrentSequence`, `LisLength`, `AttemptCount`。
    3. 將資料依時間（Timestamp）由早到晚進行排序。
    4. 回傳標準 JSON 陣列給前端。

### 前端開發 (replay_check.html)
- **資料預處理**：前端載入整包 Log 陣列後，依據時間軸的變更，動態篩選出「截至當前時間點，各隊伍最新的一筆紀錄」。
- **畫面更新機制**：將篩選出的各隊狀態丟進原有的 `updateDashboard(teams)` 函式中，驅動 DOM 結構進行 FLIP 重新排序與 LIS 反光白。

## 4. 實戰優化考量 (防呆與效能)
- **Log 資料壓縮**：若當天活動的 Log 總筆數過大，GAS 讀取可能稍慢。前端載入時可加上 Loading 動畫（如：`⏳ 正在開啟時光機，載入決賽歷史紀錄中...`）。
- **時間顯示優化**：時間軸上方不只顯示邏輯筆數（第 X 筆），應將 `Timestamp` 轉換為相較於開賽時間的相對時間（例如：`開賽後 00:23:14`），讓參賽者對時間更有體感。
